## Top 5 Findings

### 1. CRITICAL — SandboxManager / `/sandbox-toggle` deferred from spec 35 is **not covered anywhere in spec 42**

Spec 35 (`35-remote-sessions.md:18`) explicitly delegates ownership:

> "sandbox toggle / `SandboxManager` / `dangerouslySkipPermissions` × sandbox
> interaction → **42**"

and `35:623` repeats it in the cross-references section. But `42-misc.md`
contains **zero occurrences** of the strings `sandbox`, `SandboxManager`,
`/sandbox-toggle`, or any reference to `src/main.tsx:201,314-316`.
`grep -ni 'sandbox\|SandboxManager\|sandbox-toggle' 42-misc.md` returns no
hits. The §1 IN-scope list, §2 Source Map, §1 OUT-of-scope table, and §12
escalations all silently omit it.

`src/utils/sandbox/` exists (`sandbox-adapter.ts`, `sandbox-ui-utils.ts`) and
spec 42a §1 lists it under "Settings / env / sandbox" (8 files), but the
**architectural-level** SandboxManager + `/sandbox-toggle` semantics that
spec 35 escalated to 42 are unowned. Coverage gap.

### 2. HIGH — §1 IN-scope does not acknowledge the 23%-misplacement co-ownership flagged by 42a/9.5

Phase 9.5 found that bash/shell/powershell utils (28 files) belong with the
permission engine (spec 9/10), swarm utils (21) with coordinator (14/30),
computerUse (16) with tool-mcp-lsp, and model utils (17) with api (22).
42a §5 has the explicit reassignment map. Spec 42's §1 IN-scope acknowledges
*only* the buddy/upstreamproxy/native-ts/moreright/assistant/cli/services/
generated-types residuals; it does not mention the four large
co-owned subsystems at all. The §1 OUT-of-scope table lists 14 owners but
none of: 9/10 (permissions for bash), 14/30 (swarm/coordinator), 16
(tool-mcp-lsp for computerUse), 22 (api for model). Verifiable via
`grep -ni 'computerUse\|powershell\|model registry' 42-misc.md` → 0 hits.

### 3. HIGH — `bash/ast.ts` shell-trust-boundary status is invisible at the architectural level

The file is 2,679 lines (112KB). Its file-header comment is unambiguous:

> "The key design property is FAIL-CLOSED: we never interpret structure we
> don't understand. … This is NOT a sandbox."

This is the single largest trust-boundary in the entire `src/utils/` tree.
Spec 42 does not mention it. Spec 42a notes the size but defers semantics.
The architectural spec is the appropriate place for one paragraph saying
"the shell-permission-engine boundary lives at `utils/bash/ast.ts` and is
owned by spec 9/10". Not present.

### 4. MEDIUM — `ansiToPng.ts` (215KB) is not source code; spec 42 is silent on this

Verified via Read: the file header states it "blits a bundled 24×48 bitmap
font directly into an RGBA Uint8Array" — the bulk of the 215KB is the
embedded Fira Code rasterized bitmap, regenerated by
`scripts/generate-bitmap-font.ts`. Spec 42a §1 flags it ("215KB!"). Spec 42
does not categorize it as embedded asset vs source, so any LOC totals or
"survey-only" claims are misleading by ~215KB out of `src/utils/`'s ~3.93 MB
(spec 42a §0). Minor at architectural level but worth a one-line note.

### 5. MEDIUM — §A constants appendix is present but `errorIds.ts` claim is inconsistent with E_* convention

§A (line 703–719) is present as expected from Phase 10d cleanup, with 7
constants files mapped to consumer specs. Verified.

However: §A claims `errorIds.ts` `E_*` values are "obfuscated trace IDs".
The example given (`E_TOOL_USE_SUMMARY_GENERATION_FAILED = 344`) is *not*
obfuscated — the identifier name is fully descriptive. This is a wording
nit but the spec promises verbatim fidelity (§Self-check: "Every behavioral
claim cites src/<path>:<line-range>"). No line cite is given for the
"Next ID: 346" comment claim — should be tightened.

## Verdict

**APPROVE WITH ONE CRITICAL FIX REQUIRED.**

Spec 42 is structurally sound for a catch-all (good IN/OUT scope, verbatim
inlined assets in §6, residual escalations in §12, Phase 10 §A appendix
present). The leak-fidelity ratio for buddy/upstreamproxy/native-ts is high
and verifiable.

The **CRITICAL** finding (#1) is a real coverage hole created by spec 35's
unilateral delegation. Either:
(a) spec 42 must add a §X "Sandbox subsystem (delegated from 35)" with
    `SandboxManager` boot, `/sandbox-toggle` command lifecycle, and
    `src/utils/sandbox/` contents, OR
(b) spec 35's deferral pointer must be updated to a different owner (a new
    spec, or 09 permissions, since sandbox is fundamentally a permission
    primitive), OR
(c) a new tiny sub-spec 42b-sandbox.md absorbs it, parallel to 42a-utils.

Findings #2 and #3 are HIGH because spec 42 is the architectural document
for the long tail; without acknowledging the 23% co-ownership and the
shell trust-boundary at the architectural level, the spec corpus appears
self-consistent (each individual spec checks out) but is collectively
silent on the most security-critical surfaces.

## Hardest-to-Verify Claim

**`buddy/companion.ts` salt = `'friend-2026-401'` and the security claim
that "anyone can recompute the roll for a known userId" (§10).**

This is hardest because (a) it requires reading the unexported `SALT`
constant against the `roll()` flow; (b) verifying it is not a security
secret depends on the threat model (a static module constant in a leaked
bundle is by-definition not secret, but the spec's claim that "users can't
edit `config.companion` to fake a `legendary` rarity" depends on `bones`
being regenerated from `userId` on every read — which is verifiable from
the cited line range `companion.ts:121-123`, but the *implication* that
SALT prevents farming is asserted, not proven). I did not deep-read
`companion.ts:84,107-113` in this review (would have consumed 1 of the
remaining read-budget on a non-blocking claim). Flag for future revise.

Secondary hard-to-verify claim: §5.4 relay state machine — the wire-format
encode/decode (`encodeChunk`/`decodeChunk` protobuf hand-rolled at
`relay.ts:51-103`) is reproduced verbatim in §6.6 but the *semantic*
correctness against a real `UpstreamProxyChunk` protobuf schema is
out-of-leak (`bun-protobuf-gen` outputs in `types/generated/` cover
`events_mono` not upstreamproxy). Cannot fully verify from leak alone.
